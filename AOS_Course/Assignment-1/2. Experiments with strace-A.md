`strace` stands for **System Call Trace.** It trace every system call a program makes to the Linux Kernel.


Unlike `objdump`, which shows what the compiler generated, `strace` shows how the running program interacts with the Linux kernel.

**Program Execution Flow**

```text
Your Program
      │
      │  printf()
      ▼
C Standard Library (glibc)
      │
      │  write()
      ▼
Linux Kernel
      │
      ▼
Hardware
```

The program cannot communicate directly with the hardware. Whenever it needs services such as writing to the screen, reading a file, allocating memory, or creating a process, it requests them from the kernel using **system calls**.

`strace` allows us to observe these system calls.


**Compile the Program**
(already done in previous experiment)
```bash
gcc hello.c -o hello
```
- `gcc` → GNU C Compiler.
- `hello.c` → Source file.
- `-o hello` → Specifies that the output executable should be named `hello`.


**Run the Program Using `strace`**

```bash
strace ./hello
```

- `strace` → Starts tracing all system calls made by the program.
- `./hello` → Executes the `hello` program located in the current directory.

The system calls will be displayed directly on the terminal.

**Better Save the Trace to a File**

```bash
strace -o trace.txt ./hello
```

- `strace` → Starts tracing system calls.
- `-o trace.txt` → Saves the trace output into the file `trace.txt`.
- `./hello` → Executes the program.

**View the Trace File**

```bash
cat trace.txt
```

or

```bash
less trace.txt
```

- `cat trace.txt` → Displays the complete contents of the trace file.
- `less trace.txt` → Opens the trace file in a scrollable viewer, which is useful for large outputs.

## A. What system calls are invoked when you run the program?

The following system calls are observed during the execution of the program:

| System Call | Purpose |
|-------------|---------|
| `execve()` | Starts execution of the program. |
| `brk()` | Manages the process heap for dynamic memory allocation. |
| `mmap()` | Maps files and memory regions into the process's address space. |
| `access()` | Checks whether a file exists and whether the process has permission to access it. |
| `openat()` | Opens files required during program execution. |
| `read()` | Reads data from opened files. |
| `pread64()` | Reads data from a file at a specified offset without changing the file pointer. |
| `fstat()` | Retrieves information about an opened file. |
| `close()` | Closes an opened file descriptor. |
| `arch_prctl()` | Sets architecture-specific process state (used for thread-local storage). |
| `set_tid_address()` | Registers the thread ID address used by the kernel. |
| `set_robust_list()` | Registers the robust futex list for thread synchronization. |
| `rseq()` | Registers restartable sequences for efficient per-CPU operations. |
| `mprotect()` | Changes memory protection permissions. |
| `prlimit64()` | Retrieves or modifies process resource limits. |
| `munmap()` | Unmaps a memory region. |
| `getrandom()` | Obtains random bytes from the kernel. |
| `write()` | Writes output to the terminal (stdout). |
| `exit_group()` | Terminates the process. |

These system calls are responsible for loading the executable, preparing the process environment, loading shared libraries, writing the output to the terminal, and terminating the process.

## B. How many times is the `write` system call invoked? What file descriptor is being used for stdout?

The `write()` system call is invoked **once** during the execution of the program.

From the trace:

```text
write(1, "Hello, World!\n", 14) = 14
```

The first argument of `write()` is `1`, which is the file descriptor for **standard output (stdout)**.

The standard file descriptors in Linux are:

| File Descriptor | Stream |
|-----------------|--------|
| `0` | Standard Input (`stdin`) |
| `1` | Standard Output (`stdout`) |
| `2` | Standard Error (`stderr`) |

Thus, the string `"Hello, World!\n"` is written once to the terminal using file descriptor `1`.

## C. Can you identify the system calls that deal with process startup and termination?

### Process Startup

The following system calls are involved in starting the process:

- `execve()` – Loads and starts execution of the program.
- `brk()` – Initializes the process heap.
- `mmap()` – Maps the executable and shared libraries into memory.
- `openat()` – Opens required system files and shared libraries.
- `read()` and `pread64()` – Read executable and library contents.
- `fstat()` – Retrieves file metadata.
- `arch_prctl()` – Sets architecture-specific thread information.
- `set_tid_address()` – Registers the thread ID.
- `set_robust_list()` – Registers robust futex information.
- `rseq()` – Registers restartable sequences.
- `mprotect()` – Sets memory access permissions.
- `prlimit64()` – Retrieves process resource limits.

### Process Termination

The following system calls are involved in terminating the process:

- `write()` – Writes the output to the terminal before termination.
- `exit_group(0)` – Terminates the process and returns exit status `0`.

## D. What do you observe? What arguments are passed to the `write` system call?

The trace shows that the C library function `printf()` is translated into a single `write()` system call.

The observed system call is:

```text
write(1, "Hello, World!\n", 14) = 14
```

The arguments passed to `write()` are:

| Argument | Value | Description |
|----------|-------|-------------|
| File Descriptor | `1` | Standard Output (`stdout`) |
| Buffer | `"Hello, World!\n"` | The string to be written |
| Count | `14` | Number of bytes to write |

The return value is:

```text
= 14
```

which indicates that all 14 bytes were successfully written to the terminal.

## E. What files are opened and closed by the program? Can you explain why those files are being accessed?

From the trace, the program opens the following files:

1. `/etc/ld.so.cache`
2. `/lib/x86_64-linux-gnu/libc.so.6`

These files are later closed using the `close()` system call.

### Why are these files accessed?

#### `/etc/ld.so.cache`

This file contains a cache of shared libraries maintained by the dynamic linker (`ld.so`). It allows the operating system to quickly locate the required shared libraries without searching every directory listed in the library path.

#### `/lib/x86_64-linux-gnu/libc.so.6`

This is the GNU C Standard Library (`glibc`), which provides implementations of standard library functions such as `printf()`, `puts()`, `malloc()`, and many others. The program depends on this library to execute standard C functions.

After loading the required information into memory using `mmap()`, the file descriptors are closed using `close()`, since the mapped memory remains available to the process even after the files have been closed.
