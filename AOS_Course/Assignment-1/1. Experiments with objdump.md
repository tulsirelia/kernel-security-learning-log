## A) What do you observe in the disassembly? Can you identify the main function in the output? How does the disassembly relate to the original C code? 

<img width="996" height="501" alt="image" src="https://github.com/user-attachments/assets/3ff6b35c-6ae1-4d2d-9e24-55dba17001b4" />

The disassembly shows the assembly instructions generated from the compiled C program. It contains machine instructions, memory addresses, hexadecimal machine-code bytes, function labels, and CPU instructions corresponding to the original C code.

The main() function is identified by the label `<main>`:. All instructions following this label belong to the main() function.

One C statement becomes several assembly instructions. Example: 

`printf("Hello")` 

⬇️

```
lea
mov
call
```


The disassembly is the low-level representation of the original C program. Each C statement is translated into one or more assembly instructions. For example, the function prologue (`push %rbp`, `mov %rsp,%rbp`) sets up the stack frame, the `printf()` call is translated into instructions that prepare the function argument and execute a `call` instruction, and `return 0`; becomes `mov $0,%eax` followed by ret.

## B) What symbols do you see? Explain the different types of symbols listed in the table (functions, global variables, etc.).
-t → table (symbol table)
```
objdump -t hello.o
```
<img width="1007" height="362" alt="image" src="https://github.com/user-attachments/assets/81e0739a-6d1b-4b50-9844-fbaef7babd98" />

Overall Structure:
```
SYMBOL TABLE:
Address      Scope     Type      Section    Size       Name
```

**Column 1: Address**

This is the symbol's address (or offset) inside the object file.

Notice every symbol starts at `0`.

Why?

Because this is an object file (`.o`), not an executable.

The linker hasn't assigned final memory addresses yet.

Later, after linking, these addresses become actual memory locations.

**Column 2: Scope**

`l` → local symbol, only visible inside this object file, like `hello.c`, `.text`, `.rodata`

`g` → global symbol, visible to the linker. Other object files can reference it like `main`

**Column 3: Type**

This column specifies the type of symbol.

- `df` → **Debug/File Symbol**. It indicates the source file from which the object file was generated. In this case, `hello.c`.
- `d` → **Section Definition**. It represents a section present in the object file, such as `.text` or `.rodata`.
- `F` → **Function**. It indicates that the symbol is a function. Here, `main` is a function stored in the `.text` section.


**Column 4: Section**

This column indicates the section of the object file where the symbol belongs.

- `*ABS*` → **Absolute**. The symbol does not belong to any section. It is generally used for file-related symbols such as `hello.c`.
- `.text` → Contains the executable machine code of the program. Functions like `main()` are stored here.
- `.rodata` → **Read-Only Data** section. It stores constants and string literals, such as `"Hello, World!\n"`.
- `*UND*` → **Undefined**. The symbol is referenced in this object file but is not defined here. Its definition will be resolved by the linker from external libraries.


**Column 5: Size**

This column specifies the size of the symbol in bytes.

For example,

- `main` has a size of `0x1e` (30 bytes), which means the compiled machine code for the `main()` function occupies 30 bytes.
- Section symbols and undefined symbols usually have a size of `0` because they do not represent executable code or allocated data in this object file.


**Column 6: Symbol Name**

This column contains the actual name of the symbol.

From the output:

- `hello.c` → Indicates that the object file was generated from the source file `hello.c`.
- `.text` → Represents the executable code section.
- `.rodata` → Represents the read-only data section that stores constants and string literals.
- `main` → The user-defined function in the program. It is a global function stored in the `.text` section.
- `puts` → An external library function used by the program. It appears as an undefined (`*UND*`) symbol because it is not implemented in this object file. During the linking stage, the linker resolves it using the C standard library (`libc`). Although the source code uses `printf("Hello, World!\n")`, modern GCC optimizes it to `puts("Hello, World!")` since there are no format specifiers in the string.
